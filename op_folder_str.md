# Output Folder Structure Documentation

This document describes the output folder structures generated by `generate_contracts.py` and `compile_and_analyze.py`.

---

## Overview

- **`generate_contracts.py`**: Generates Solidity smart contracts using LLM models (Ollama)
- **`compile_and_analyze.py`**: Compiles generated contracts and performs security analysis using Slither

These scripts work in sequence, with `compile_and_analyze.py` depending on outputs from `generate_contracts.py`.

---

## 1. Output Structure from `generate_contracts.py`

### Default Output Directory
- **Location**: `output/`

### Main Folder Structure

```
output/
├── generation_results.json          # Complete generation results
├── generation_summary.json          # Summary with statistics and folder mapping
├── [model_1]/                       # Folder for each model (sanitized name)
│   ├── [prompt_1]/                  # Folder for each prompt (sanitized name)
│   │   ├── 0.sol                    # Solidity code for iteration 0
│   │   ├── 0.txt                    # Raw LLM response for iteration 0
│   │   ├── 1.sol                    # Solidity code for iteration 1
│   │   ├── 1.txt                    # Raw LLM response for iteration 1
│   │   ├── ...
│   │   └── [NUM_ITERATIONS-1].sol   # Final iteration
│   ├── [prompt_2]/
│   │   ├── 0.sol
│   │   ├── 0.txt
│   │   └── ...
│   └── [prompt_n]/
├── [model_2]/
│   ├── [prompt_1]/
│   └── ...
└── [model_n]/
```

### Key Parameters

| Parameter | Default | Description |
|-----------|---------|-------------|
| `NUM_ITERATIONS` | 5 | Number of iterations per prompt |
| `MODELS` | Various LLM models | List of models (gemma3:27b, gpt-oss:20b, mistral-small:24b, etc.) |
| `OLLAMA_TIMEOUT` | 180s | Timeout for API calls (seconds) |
| `MAX_RETRIES` | 2 | Maximum retry attempts for failed requests |

### Output Files Generated

#### 1. `generation_results.json`
- **Purpose**: Complete results for all generated contracts
- **Structure**:
  ```json
  {
    "[model_name]": {
      "[prompt_name]": {
        "iteration_0": {
          "response": "...",                    // Raw LLM response
          "solidity_code": "...",               // Extracted Solidity code
          "timing": {
            "generation_seconds": 2.5,
            "total_duration_ns": 2500000000
          },
          "error": null
        },
        "iteration_1": { ... },
        ...
      }
    }
  }
  ```

#### 2. `generation_summary.json`
- **Purpose**: Summary statistics and folder mapping
- **Structure**:
  ```json
  {
    "timestamp": "2026-06-19T12:00:00",
    "statistics": {
      "total_prompts": 10,
      "successful_generations": 150,
      "failed_generations": 0,
      "total_time": 1234.56,
      "model_times": {
        "gemma3:27b": {
          "seconds": 400.12,
          "minutes": 6.67
        }
      }
    },
    "folder_mapping": {
      "[model_name]": {
        "folder_name": "sanitized_model_name",
        "prompts": {
          "[prompt_name]": {
            "folder_name": "sanitized_prompt_name",
            "path": "sanitized_model_name/sanitized_prompt_name"
          }
        }
      }
    },
    "configuration": {
      "models": [...],
      "iterations": 5,
      "timeout": 180
    }
  }
  ```

### Folder Naming Convention

- Model names and prompt names are **sanitized** by replacing invalid Windows file path characters:
  - `< > : " / \ | ? *` → `_`
- Example: `gpt-oss:20b` → `gpt-oss_20b`

- Requires `generation_results.json` and `generation_summary.json` from `generate_contracts.py`

#### 3. `cp_detailed_summary.txt`
- **Purpose**: Comprehensive human-readable summary report
- **Contents**:
  - Analysis timestamp
  - Per-model statistics:
    - Generation statistics (successful/failed/success rate)
    - Compilation statistics
    - Security analysis (clean contracts, vulnerability breakdown)
    - Top 10 most common malign vulnerabilities
    - Per-prompt summary with vulnerability counts
  - Note: Excludes Informational and Optimization issues, focuses only on security-critical vulnerabilities (High/Medium/Low)

#### 4. `cp_quick_summary.txt`
- **Purpose**: Concise summary statistics
- **Contents** (per model):
  - Average generation time
  - Compilation rate (successful/total)
  - Clean contracts count
  - Vulnerability breakdown:
    - High severity count
    - Medium severity count
    - Low severity count

## 3. Vulnerability Severity Levels

From Slither analysis, vulnerabilities are categorized by severity:

| Severity | Description | Included in Reports |
|----------|-------------|-------------------|
| **High** | Critical security issues | ✓ Malign only |
| **Medium** | Significant vulnerabilities | ✓ Malign only |
| **Low** | Minor security concerns | ✓ Malign only |
| **Informational** | Informational findings | ✗ Excluded from malign reports |
| **Optimization** | Gas optimization suggestions | ✗ Excluded from malign reports |

**Malign vulnerabilities**: High + Medium + Low (excludes Informational/Optimization)

## 4. Timing and Performance Metrics

### Generation Phase (`generate_contracts.py`)
- **Metrics tracked**:
  - Per-contract generation time (seconds)
  - Total duration from Ollama (nanoseconds)
  - Per-model execution time
  - Total execution time

### Compilation and Analysis Phase (`compile_and_analyze.py`)
- **Metrics tracked**:
  - Compilation time per contract
  - Slither analysis time per contract
  - Total processing time
  - Average timings across all contracts

## 5. Example Usage

### Generate Contracts
```python
from generate_contracts import ContractGenerator

generator = ContractGenerator(output_dir="output")
generator.run_generation("prompts_final.csv")
```

**Output**: `output/` folder with model subfolders, .sol/.txt files, and JSON summaries

### Compile and Analyze
```python
from compile_and_analyze import CompilerAndAnalyzer

analyzer = CompilerAndAnalyzer("output")
analyzer.load_generation_results()
analyzer.compile_and_analyze_all()
analyzer.analyze_results()
analyzer.save_results()
analyzer.generate_summary_report()
```

**Output**: Additional JSON analysis files and text summaries in `output/` folder

## 7. Notes on File Organization

1. **Sanitization**: Both model and prompt folder names are sanitized to ensure Windows compatibility
2. **Iterative Structure**: Each prompt has multiple iterations (default: 5) to test variability
3. **Dual Output**: Each contract is saved in two formats:
   - `.sol`: Extracted Solidity code (clean format)
   - `.txt`: Raw LLM response (for debugging/reference)
4. **JSON Storage**: All structured data is stored as JSON for programmatic access
5. **Human-Readable Reports**: Text summaries are generated for easy review and reporting
